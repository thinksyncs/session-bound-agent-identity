\section{Session-Bound Agent Identity
Profile}\label{session-bound-agent-identity-profile}

Draft v0.5-review3

Copyright (c) 2026 ToppyMicroServices OÜ

Author: Akira Okutomi

SPDX-License-Identifier: Apache-2.0

\subsection{Abstract}\label{abstract}

This profile binds upper-layer identity and authorization material to
one accepted TLS 1.3 session and to the attestation facts accepted for
that session. It is not a TLS extension and it does not depend on AGTP,
Agent2Agent, or any other application protocol for its trust model.

The verifier accepts a peer only when the authenticated identity,
session binding, freshness state, attestation state, and local policy
all describe the same intended session, platform, service, agent, task,
and authority boundary.

\subsection{1. Status and scope}\label{status-and-scope}

This is a repository security-hardening profile. It does not define a
new TLS handshake, attestation evidence format, identity provider, or
application core protocol. It composes existing mechanisms and adds
fail-closed acceptance rules.

The key words \textbf{MUST}, \textbf{MUST NOT}, \textbf{REQUIRED},
\textbf{SHALL}, \textbf{SHALL NOT}, \textbf{SHOULD}, \textbf{SHOULD
NOT}, \textbf{RECOMMENDED}, \textbf{NOT RECOMMENDED}, \textbf{MAY}, and
\textbf{OPTIONAL} are interpreted as in BCP 14, RFC 2119, and RFC 8174
when they appear in capitals.

A rule belongs in this profile when omitting it could cause one of these
failures:

\begin{itemize}
\tightlist
\item
  accepting a binding from the wrong live TLS or attestation session;
\item
  accepting the right platform for the wrong service, tenant, agent,
  task, or authority boundary;
\item
  accepting replayed, stale, missing, downgraded, or partly verified
  state;
\item
  treating peer-selected metadata as verifier policy;
\item
  reusing a security result outside the session or request for which it
  was produced.
\end{itemize}

Out of scope are generic OAuth, OIDC, JWT, JWS, CWT, COSE, DID, VC, TLS,
and attestation behavior except where this profile uses them for
fail-closed acceptance.

Normative requirements apply to profile behavior, not to a package name
or library layout. Implementation-status text is descriptive unless it
uses BCP 14 keywords.

\subsection{2. Terms}\label{terms}

\textbf{Session-Bound Agent Identity Profile} means an
application-profile acceptance gate over ordinary TLS 1.3,
post-handshake platform attestation, session binding, authenticated
identity material, and verifier-local policy.

\textbf{Identity Grant} means an authority statement issued by a Manager
or another locally trusted policy authority. It authorizes upper-layer
identity, task, and authorization values. It does not prove presence on
the current TLS session.

\textbf{Session Binding Statement} means a holder-of-key proof that
binds one verified Identity Grant to one accepted TLS or
exported-authenticator session. It does not authorize service, tenant,
task, scope, resource, or capability values.

\textbf{Expected values} are verifier-local policy inputs.
\textbf{Observed values} are extracted from authenticated grants,
session-bound statements, attestation claims, trusted manifests, or
locally derived request state. Observed values do not become expected
values without a trusted local policy decision.

\textbf{Acceptance decision} means returning a peer identity or
authorization result to the application as profile-authenticated. Replay
state for one-shot bindings is committed before this decision is
returned.

The older shorthand \texttt{aTLS} is reserved for existing CoCos
implementation code, package names, and historical text. AGTP-named
packages, tests, claim aliases, and notes may remain in the repository.
In this profile, AGTP is an implementation/reference namespace only; it
is not an application-protocol dependency or authority source.

\subsection{4. Threat model and layer
model}\label{threat-model-and-layer-model}

The attacker can observe, replay, reorder, relay, or substitute network
and application messages. The attacker can supply malicious peer
metadata and can run another Agent on the same host or deployment
environment.

The lower layers are assumed to validate TLS 1.3, exported
authenticators, and platform evidence correctly. This profile defines
what must be bound to those facts before the application treats the peer
as the intended actor.

Out of scope are a fully compromised verifier process or local policy
store, a malicious trusted Manager, compromise of all local secret
storage, denial of service, and side channels outside the
identity-binding path.

These keys may be related by deployment policy, but they are not

Manager or policy-authority signing keys. Endpoint keys are valid for
session binding only when the verified grant or local policy explicitly
authorizes that use.

The protected-header \texttt{kid} is only a key-selection hint.
Acceptance also depends on issuer, audience, algorithm allow-list, key
type, key use, key status, profile version, token type, time validity,
revocation state, and local policy. The \texttt{none} algorithm is
forbidden.

\subsubsection{5.1 Identity Grant}\label{identity-grant}

An Identity Grant authorizes upper-layer semantics. It is signed by a
Manager or trusted policy authority and identifies the Agent
confirmation key, usually through a confirmation-key claim such as
\texttt{cnf.kid}.

The grant may carry L3 through L6 fields, including service, tenant,
deployment, workload, Agent, task, delegation, canonical intent or
capability references, scopes, resources, and authorization details. The
Agent is not the authority for these values. They are accepted only
after grant verification and local policy comparison.

The normative token guard names are \texttt{profile\_type} and
\texttt{profile\_version}. Legacy adapters MAY expose
\texttt{agtp\_type} and \texttt{agtp\_version} as compatibility aliases,
but a verifier MUST NOT treat AGTP names as the authority model.

\subsubsection{5.2 Session Binding
Statement}\label{session-binding-statement}

A Session Binding Statement proves that the holder of the confirmation
key bound the verified grant to the accepted TLS or
exported-authenticator session. It contains at least:

\begin{itemize}
\tightlist
\item
  \texttt{profile\_type} and \texttt{profile\_version};
\item
  \texttt{aud}, \texttt{jti}, \texttt{iat}, and \texttt{exp};
\item
  \texttt{grant\_hash} over the exact signed grant bytes;
\item
  \texttt{leaf\_public\_key\_sha256};
\item
  \texttt{tls\_exporter\_sha256};
\item
  \texttt{request\_context\_sha256};
\item
  \texttt{nonce};
\item
  \texttt{attestation\_binder\_sha256} when accepted
  attestation-to-channel evidence is present.
\end{itemize}

For the initial encodings, \texttt{grant\_hash} is computed over the
exact signed bytes:

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{grant\_hash = SHA{-}256("sbaip.identity{-}grant.jwt.v1" || NUL || exact{-}signed{-}grant{-}bytes)}
\NormalTok{grant\_hash = SHA{-}256("sbaip.identity{-}grant.cwt.v1" || NUL || exact{-}signed{-}grant{-}bytes)}
\end{Highlighting}
\end{Shaded}

A verifier MUST NOT compute \texttt{grant\_hash} by reserializing JSON
in the acceptance path. Canonical CBOR/COSE is acceptable only when the
encoding profile fixes the canonical form.

\subsection{6. Wallet presentation
model}\label{wallet-presentation-model}

A wallet can be used as the holder-side presentation component. It can
store Identity Grants, verifiable credentials, DID references, reference
tokens, and confirmation-key material. It can also protect the Agent
binding key and create or present a Session Binding Statement after a
verifier challenge.

A wallet is not a trust root for this profile. Wallet UI labels, display
names, card fields, credential nicknames, and peer-supplied metadata are
not expected policy. A wallet presentation is accepted only if the
verifier can validate the issuer or Manager authority, holder key,
session binding, freshness, replay state, and local policy.

Pre-signed wallet presentations are not session-bound by themselves. A
presentation that lacks the current \texttt{aud}, \texttt{nonce},
\texttt{grant\_hash}, \texttt{request\_context\_sha256}, and TLS
exporter binding MUST NOT be accepted as proof for the current session.

Selective disclosure and reference tokens are useful privacy tools. They
do not change the acceptance rule: the disclosed or referenced
attributes still need a trusted issuer, a holder proof, a fresh session
binding, and local policy comparison.

\subsection{7. L2 binding construction}\label{l2-binding-construction}

This section fixes the direct-Agent L2 construction. A verifier MUST NOT
replace these inputs with peer-selected labels, inferred context,
display names, or reserialized semantic metadata.

Inputs:

\begin{itemize}
\tightlist
\item
  \texttt{tls\_connection}: the accepted TLS 1.3 connection;
\item
  \texttt{exporter\_label}: ASCII \texttt{Attestation} for the legacy v1
  construction;
\item
  \texttt{leaf\_spki}: DER SubjectPublicKeyInfo of the accepted leaf
  certificate or exported-authenticator certificate;
\item
  \texttt{H}: the TLS 1.3 authenticator hash for the negotiated cipher
  suite;
\item
  \texttt{role}, \texttt{protocol\_id}, \texttt{aud},
  \texttt{grant\_hash}, \texttt{task\_context}, and
  \texttt{verifier\_nonce\_or\_attempt\_id} from verifier-local state.
\end{itemize}

New profiles SHOULD use a namespaced exporter label. The v1 label
remains fixed for compatibility. A peer-selected exporter label is never
accepted.

The context bytes are fixed by \texttt{sbaip\_context\_v1}:

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{field(name, value) = u16be(len(name)) || name || u32be(len(value)) || value}

\NormalTok{context = "SBAIP{-}CONTEXT{-}v1" || NUL ||}
\NormalTok{          field("role", role) ||}
\NormalTok{          field("protocol\_id", protocol\_id) ||}
\NormalTok{          field("aud", aud) ||}
\NormalTok{          field("grant\_hash", raw\_32\_byte\_grant\_hash) ||}
\NormalTok{          field("task\_context", task\_context) ||}
\NormalTok{          field("verifier\_nonce\_or\_attempt\_id", verifier\_nonce\_or\_attempt\_id)}
\end{Highlighting}
\end{Shaded}

\texttt{role}, \texttt{protocol\_id}, \texttt{aud}, and
\texttt{task\_context} are UTF-8 bytes after verifier-local
canonicalization. \texttt{grant\_hash} is the raw 32-byte digest, not a
hex string. \texttt{task\_context} is itself a length-delimited
canonical byte string when it contains multiple fields.

Construction:

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{EKM = TLS{-}Exporter(tls\_connection,}
\NormalTok{                   label = "Attestation",}
\NormalTok{                   context = context,}
\NormalTok{                   length = 32)}

\NormalTok{leaf\_public\_key\_sha256     = SHA{-}256(leaf\_spki)}
\NormalTok{tls\_exporter\_sha256        = SHA{-}256(EKM)}
\NormalTok{request\_context\_sha256     = SHA{-}256(context)}
\NormalTok{attestation\_binding        = H(leaf\_spki || EKM)}
\NormalTok{attestation\_binder\_sha256  = SHA{-}256(attestation\_binding)}
\end{Highlighting}
\end{Shaded}

The Session Binding Statement carries the SHA-256 values, normally
serialized as lowercase hexadecimal without a \texttt{sha256:} prefix in
the current Go adapters.

\texttt{leaf\_public\_key\_sha256} confirms the endpoint key, but it is
not session unique. The primary session binding is the TLS exporter
under the accepted context, plus the grant hash, audience, nonce,
attestation binder when present, and replay state.

Attestation evidence or attestation results must be appraised against
the same \texttt{attestation\_binding}. For evidence formats that expose
report data or nonce, the lower layer derives:

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{report\_data = SHA{-}512(attestation\_binding)}
\NormalTok{nonce       = SHA{-}256(EKM)}
\end{Highlighting}
\end{Shaded}

A verifier compares \texttt{tls\_exporter\_sha256} and
\texttt{request\_context\_sha256} with its own accepted values. Missing
or mismatched mandatory fields fail closed.

\subsection{8. Verification and policy
model}\label{verification-and-policy-model}

Acceptance has two phases.

\textbf{Authentication phase}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Verify the Identity Grant under a trusted Manager or policy-authority
  key.
\item
  Check issuer, audience, profile guard, algorithm, key status,
  \texttt{iat}, \texttt{exp}, grant ID, and required authorization
  fields.
\item
  Verify the Session Binding Statement under the Agent confirmation key
  or an explicitly authorized endpoint key.
\item
  Check \texttt{grant\_hash}, \texttt{aud}, endpoint-key hash, TLS
  exporter hash, request context hash, attestation binder, nonce, and
  replay state.
\item
  Build an internal observed assertion only from verified material.
\end{enumerate}

\textbf{Policy phase}:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Load verifier-local expected values for required layers.
\item
  Reject missing, ambiguous, non-canonical, or peer-supplied expected
  values.
\item
  Compare each observed L3 through L6 value with local expected policy.
\item
  Enforce L6 set semantics. The default is exact set equality.
  \texttt{contains\_all} is allowed only when local policy explicitly
  permits surplus observed values for that endpoint, task, Agent, and
  resource.
\end{enumerate}

Neither an Identity Grant alone nor a Session Binding Statement alone is
enough. Sender-constraining proves possession and freshness; it does not
authorize surplus scopes, resources, capabilities, or authorization
details.

\subsection{9. Canonical policy values}\label{canonical-policy-values}

Decision-sensitive semantic values must be canonical before acceptance.
Examples include:

\begin{itemize}
\tightlist
\item
  \texttt{ontology\_id} for the vocabulary or policy namespace;
\item
  \texttt{intent\_ref} for the intended task or action family;
\item
  \texttt{capability\_ref} for a resource-action class.
\end{itemize}

Receivers MUST NOT repair peer-provided aliases, display labels, natural
language phrases, case variants, URI variants, or model interpretations
during the final acceptance path. Alias resolution belongs to the
Manager, trusted registry, or local policy engine before issuance or
comparison.

An alias is acceptable only when it resolves to exactly one canonical
reference under a pinned registry namespace and version. Missing,
ambiguous, deprecated, or unsupported registry data fails closed.

issuer trust, confirmation-key authorization, replay, or local-policy
requirements.

\subsection{11. Freshness, replay, rotation, and
revocation}\label{freshness-replay-rotation-and-revocation}

The accepted assertion expiry is the earliest applicable expiry among:

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{grant exp,}
\NormalTok{Session Binding Statement exp,}
\NormalTok{attestation{-}result exp,}
\NormalTok{evidence challenge lifetime,}
\NormalTok{attestation collateral or TCB lifetime,}
\NormalTok{replay{-}cache TTL,}
\NormalTok{local policy maximum TTL}
\end{Highlighting}
\end{Shaded}

A missing required freshness policy value fails closed. Evidence without
a trusted timestamp is acceptable only for the current challenge and
current authentication attempt.

Replay protection is keyed at least over:

\begin{Shaded}
\begin{Highlighting}[]
\NormalTok{grant\_hash || aud || tls\_exporter\_sha256 || request\_context\_sha256 || nonce}
\end{Highlighting}
\end{Shaded}

When available, the key also includes the binding statement ID,
task-binding value, evidence nonce, verifier challenge, or
attestation-result identifier.

\begin{Shaded}

stale, wrong-use, or ambiguous keys fail closed. Deployments that need
early invalidation must support grant revocation, Manager-key
revocation, and Agent-binding-key revocation separately.

\subsection{12. Cache, privacy, and
diagnostics}\label{cache-privacy-and-diagnostics}

Response caching and security-state caching are different. Verified
grants, Session Binding Statements, attestation evidence, authorization
decisions, and verification results MUST NOT be cached as acceptance
evidence for a later session or request.

The default for profile-sensitive HTTP responses is \texttt{no-store}. A
shared cache may store a response only when it is invariant across Agent
ID, principal, tenant, authority scope, policy grant, mTLS identity,
attestation result, session binding, task state, and other
verifier-local policy inputs. \texttt{Vary} is not enough when the
decision depends on non-header security state.

Identity-binding data can create correlation. Deployments SHOULD prefer
audience-scoped identifiers, short lifetimes, selective disclosure,
reference tokens, and redacted logs. Short lifetime alone is not a
privacy control.

Diagnostics preserve layer, field, and error class, but they do not echo
raw peer values. Validators reject invalid UTF-8, C0/C1 controls, CR,
LF, DEL, and HTML delimiter code points \texttt{\textless{}},
\texttt{\textgreater{}}, \texttt{\&}, \texttt{"}, and
\texttt{\textquotesingle{}} in decision-sensitive string fields unless
the field profile explicitly permits and canonicalizes them. Individual
values are limited to 1024 bytes and set-like fields to 128 values
unless a stricter deployment policy applies.

\subsection{13. Deployment topologies}\label{deployment-topologies}

\subsubsection{13.1 Direct-Agent mode}\label{direct-agent-mode}

In direct-Agent mode, the Agent terminates TLS and signs the Session
Binding Statement with the confirmation key named by the verified
Identity Grant, or with an endpoint key explicitly authorized by the
grant or local policy.

\subsubsection{13.2 Gateway-routed mode}\label{gateway-routed-mode}

Gateway-routed mode is a separate profile. Gateway session binding
proves the gateway endpoint, not the final Agent process. The relying
party accepts the final Agent only after a Gateway Route Assertion
passes local route policy, freshness, replay, and any required
final-Agent holder-of-key check.

The Gateway Route Assertion binds at least the gateway identity,
relying-party audience, grant hash, gateway session-binding hash, route
ID, tenant or authority partition, policy ID, target Agent,
request-context hash, nonce, and expiry. Missing, stale, replayed,
wrong-route, wrong-tenant, wrong-policy, wrong-task, or wrong-Agent
assertions fail closed.

\texttt{pkg/agtp/gatewayroute} is an implementation/reference namespace
for local route-assertion validation. JWT/JWS and CWT/COSE
route-assertion adapters may exist under AGTP-named packages, but the
AGTP name is not normative for this profile.

\subsection{14. Implementation status}\label{implementation-status}

The reusable comparator lives in \texttt{pkg/atls/identitypolicy}. It
compares a session-bound observed assertion with verifier-local expected
policy. It does not parse wire tokens, verify signatures, choose keys,
rotate keys, check revocation, or own distributed replay storage.

\texttt{identitypolicy.NewAssertionFromSessionBinding} builds an
internal assertion after external identity material has already been
authenticated. The statement must already be tied to the grant,
audience, allowed confirmation key, minimum session-binding fields, and
replay policy.

The client transport calls identity-policy validation after exported
authenticator and attestation validation, but before returning the
accepted TLS connection to the caller. Callers may provide an
\texttt{ObservedIdentity} callback or pass a verified Identity Grant,
verified Session Binding Statement, and replay cache through
\texttt{atls.ClientConfig}.

\texttt{pkg/agtp} provides concrete JWT/JWS and CWT/COSE adapters. The
package name is historical/reference naming. The adapters verify grants
and binding statements using locally configured issuer, audience,
signing methods, protected \texttt{kid}, and key lookup policy. They do
not choose trusted Manager keys, rotate keys, define revocation policy,
define deployment policy, or provide distributed replay storage.

\texttt{VerifySessionIdentityJWT} is the fail-closed direct JWT/JWS
gate: it verifies the Manager-signed grant, verifies the Session Binding
Statement, checks binding-signer authorization, compares session binding
and local policy, and then commits the replay nonce before returning
acceptance.

\subsection{15. Evaluation boundary and conformance
checklist}\label{evaluation-boundary-and-conformance-checklist}

Tests and local harnesses are useful but not enough for the full
security claim. The claim is receiver-verifiable linkage: the verified
grant, Session Binding Statement, accepted session, attestation result,
replay state, and local expected policy all describe the same intended
interaction.

An implementation claiming this profile should evaluate relay, borrowed
attestation, token substitution, multiple tasks on one TLS connection,
HTTP/2 or gRPC connection reuse, TLS resumption, 0-RTT behavior,
distributed replay-cache races, gateway route confusion, malformed
JWT/CWT/COSE inputs, duplicate JSON members, Unicode edge cases, and the
invariant that a grant is accepted only with the intended binding,
session, and local policy.

Minimum conformance:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  Identity Grant and Session Binding Statement authorities are
  separated.
\item
  The statement binds the exact signed grant bytes through
  \texttt{grant\_hash}.
\item
  L2 uses the fixed exporter label and \texttt{sbaip\_context\_v1} bytes
  for this profile version.
\item
  The verifier compares endpoint key, exporter hash, context hash,
  nonce, and attestation binder when present.
\item
  Replay state is atomic and committed before acceptance is returned.
\item
  Required L3 through L6 values come from local expected policy, not
  peer metadata.
\item
  Decision-sensitive semantic references are canonical and
  registry-scoped.
\item
  JWT/CWT/COSE parsing is fail-closed for type, algorithm, key,
  audience, duplicate-field, nesting, and critical-header ambiguity.
\item
  Accepted assertion expiry is the earliest applicable component expiry.
\item
  Gateway mode authenticates the gateway endpoint and gateway-to-Agent
  route separately.
\end{enumerate}

\subsection{Appendix B. Packaging and trust-flow
note}\label{appendix-b.-packaging-and-trust-flow-note}

A deployment may package the Manager-signed Identity Grant inside or
alongside the Agent-signed Session Binding Statement. Packaging does not
merge authorities. The verifier still authenticates the Manager
signature, the Agent binding signature, the exact grant hash, the
current session binding, replay state, and local policy.

Moving semantic grant fields directly into an Agent-signed Session
Binding Statement changes the trust model by making the Agent the
authority for service, tenant, task, scope, resource, and capability
claims. Profiles that need Manager or policy-authority authorization
MUST NOT do that.

A single Manager-signed token that also includes session-binding values
is possible only when the Manager participates in, or receives
trustworthy evidence of, the per-connection binding. That design moves
freshness and replay responsibility to the Manager path.
